    在GUI Agent的使用过程中，从 OmniParser 落地，走向 Florence-2模型微调：我们如何把“能看懂UI”变成“能稳定交付”。

    一、前情提要：我们面临的质量问题

    1.1 GUI Agent的引入

    在测试领域，长期以来，提到UI自动化测试，我们自然会想到开发效率低、维护成本高、可扩展性低这些痛点，因此借助大模型飞速发展的加持，测试领域也迎来 新的范式，GUI Agent，即：像人一样能够看懂屏幕(Web端/移动端)，操作UI的Agent。

    过去一年里，我们在实际业务中落地了PageEyes Agent，它是一个基于Pydantic AI 开发的Python UI自动化 Agent 框架。 其中，测试人员只需要输入自然语言指令，OmniParser 负责页面感知，Agent负责解析指令执行UI自动化测试。

    更多详见: - 基于AI的UI自动化提效：PageEyes Agent应用探索 - 基于 Pydantic AI + OmniParser 从零打造 GUI Agent

    如果只看 Demo，OmniParser的 “屏幕理解”方案非常完备：

    • 截一张图，它能解析界面结构；
    • 元素有框、有文本、有图标描述；
    • 再配合大模型，测试指令就能被翻译成点击、输入、滚动。

    但是进入真实业务场景，我们的配置是小时级别的任务巡检，随着各业务的交互用例持续录入，很快迎来新的挑战：回归核心长用例、高频连续运行、token成本飙升。 而从Demo到质量保障工程化的路上，我们面临一个残酷但合理的要求：

    你可以偶尔聪明，但你必须长期稳定。

    OmniParser中，用于图像识别的模块为Florence-2模型，OmniParserV2版本提供了官方微调后的Florence-2-base-ft模型， 应用这个版本后发现，它提供的视觉感知能力并不足以满足目前的业务诉求：缺乏图标的语义识别。

    1.2 图标语义识别为什么成为“质量交付”的瓶颈？

    在 UI 自动化测试中，文本可见时很好办：OCR + 语义匹配即可。

    但真实业务尤其是移动端业务中，有部分操作并不靠文字，举个例子，下图是QQ音乐客户端内的腾讯音乐榜单页面：

    大量交互图标给出的是基础形态描述，对于模型来说提升了理解难度，对于强依赖无文字图标的交互用例来说，用例基本无法执行！

    从 OmniParser 的完整链路来看，页面感知通常包含 元素定位 → 区域裁剪 → 图像语义描述 → 语义到动作映射 等多个阶段。 在我们的问题定位中，元素框定位和裁剪整体是稳定的，失败主要集中在 “图像语义描述 → 业务语义映射” 这一环节。 也就是说，系统能够稳定找到“这是一个可交互区域”，但 无法稳定回答“它在业务上代表什么”。

    该问题在无文字图标场景下被显著放大，最终成为影响自动化用例长期稳定运行的主要瓶颈。带来的问题有：

    1、错报：把图标A当成图标B，点错地方，流程跑偏，导致“用例执行成功但校验无效”。 2、漏报：认不出关键按钮，用例中断，产生大量噪声失败。

    所以，问题来到如何让Florence-2的视觉感知更贴近质量交付：

    如何让系统稳定建立 视觉符号 --> 业务语义 的映射？

    二、图标映射业务语义：用什么方式解决？

    在决定微调之前，我们首先尝试了成本更低的 Prompt Engineering 。 Florence-2 支持多任务 Prompt（如 <CAPTION>, <MORE_DETAILED_CAPTION>）。 但是实际测试表明，问题并不在于描述是否足够“详细”，而在于描述越详细 并不等同于 越方便Agent理解，自由文本描述本身并不是一个适合高频自动化执行系统的表示。

    对于 GUI Agent 而言，下游系统需要的是 短、稳定、可枚举的语义信号，而非开放式的自然语言描述。 即便通过 Prompt Engineering 能在个别样例上生成更贴近业务的描述，也难以保证在长期、高频运行中保持一致性和可控性。

    此外，受限于视觉模型的上下文长度，通过塞入大量示例的 Few-shot方案也根本不现实。

    既然 Prompt 优化走不通，考虑了其他替代方案： * 换模型：简单粗暴的方式，换一个更强大的视觉模型或者多模态模型。 * 图标库匹配：收集业务语境所需的icon，维护自己的图标库，在页面感知时通过CV的方式进行匹配。

    但是，从质量保障工程角度，这些方案同样面临新的挑战：

    • 可解释：失败时是否能知道它为什么失败？
    • 可控：成本有限，能否通过数据或配置让它稳定变好？
    • 可回归：改动后，能用标准化评估判断模型真的有变好吗？
    • 可演进：业务迭代、页面主题变化、新 icon 出现时，系统能否持续自我修复？

    很明显，换多模态模型或许省事，但对于需要权衡成本与用例高频运行的团队来说不行。

    而对于图标的特征匹配，也存在一些问题，例如： 图标在深色模式、浅色模式、描边/填充风格之间切换时，视觉差异可能远大于直觉。

    业务上线新的icon时，能否识别是未知的，需要不断对匹配图标库更新。

    在UI里，可能存在相似图标， 比如：关闭（X） vs 清除输入（X），分享 vs 转发，这类icon差异小、语义却不一样，图形特征信息不足以区分它们。

    最后，回到微调Florence-2模型上来，结合质量保障的要求，可以提炼出几点需求： - 输出可控的短文本，便于大模型理解 - 应具备表达“不是图标”的能力，对于不认识的图标应当拒绝 - 对开放集友好，对于新icon也应当具备一定泛化能力

    三、Florence-2训练集构建：如何实现高质量训练集？

    正所谓"Garbage in, garbage out"，高质量的训练集是关键。

    在 Florence-2 中，图像首先经视觉编码器提取高层视觉特征，再由任务 token（如 <CAPTION>）驱动文本解码过程生成对应描述。 我们在微调中关注的并非模型的底层视觉感知能力，而是如何让已有视觉特征在解码阶段更稳定地对齐到业务可用的语义空间。

    微调格式上，Florence-2训练集构建的思路是：给定任务 token，生成一段文本。 OmniParser使用的是<CAPTION>任务，所以微调时采用同样指令。对于图像和文本描述组成的任务token，从三个类型入手： - 业务核心图标 - 通用图标 - 负样本（各种“不是图标”的图片）

    3.1 痛点：业务核心图标收集成本高

    作为业务里最关键、最常见、最需要稳定的语义映射图标，这类数据是最高优保障集。核心图标收集主要来自两个渠道：实际业务截图、视觉设计稿，由于 人工裁剪小图标加标注费时费力，我们采用了yolo数据集构建的方式，利用工具labelimg对少量业务图片打标，维护原始图片集、YOLO标签集、图标位置集，并建立起YOLO标签集-->Florence-2 Caption集的映射。

    如果仅仅收集核心图标，训练素材量级是不够的。为了丰富训练集，使用了常规的图像增强方式，比如亮度、对比度、分辨率、旋转、裁剪等（慎用旋转，可能导致模型分不清方向），

    此外，我们先后尝试过更复杂的增强，比如图标进行二值化、图标轮廓提取， 本地测试了批量图标后发现效果不佳，可能出现图片纹理丢失等情况，无法作为训练样本使用。

    因此建立了三类增强流水线，简单随机增强、自适应阈值增强、Canny边缘增强。

    简单随机增强：模拟真实场景中的显示亮度、轻微形变、背景干扰和遮挡。 - 色彩扰动：对亮度、对比度、饱和度进行随机调整，模拟不同屏幕显示效果。 - 几何微调：进行轻微的平移、缩放和旋转（严格限制角度，避免方向混淆），模拟不同分辨率和布局下的图标形态。 - 质感模拟：引入高斯模糊或噪声，模拟低质量图片或压缩失真。 - 遮挡模拟：随机挖孔，强迫模型关注图标的局部特征，提高抗遮挡能力。

    自适应阈值增强：针对颜色复杂的图标，模拟二值化渲染效果。 - 局部自适应：根据局部像素分布提取纹理细节，更贴近 UI 渲染逻辑。 - 特征互补：保留了更多面状特征而非仅线条特征，丰富了模型的视觉输入空间。

    Canny边缘增强：让模型学习形状特征，提高对简笔画或线稿的泛化能力。 - 边缘提取：使用Canny算法提取图标轮廓，并合并双重轮廓，保留核心线条。 - 轻微扰动：辅以轻微的几何变换和色彩抖动，防止模型对特定线条形态过拟合。

    样本增强实战经验

    实际上，Canny边缘增强提取的线条并不流畅，可能包含断裂或噪点。我们期望此类样本类似 Dropout 的作用，可以引入特征噪声来防止过拟合。 但如果噪声过大可能导致整张图片失去形态特征，因此，Canny边缘增强设置了最小概率，绝大部分增强样本将通过前两个场景生成。

    这三类增强策略：简单增强增加场景多样性，自适应阈值模拟渲染逻辑，Canny辅助强化形态特征，共同提升核心图标的丰富度。以下为增强后图标示例：

    3.2 痛点：如何解决“开放集友好”问题

    如何在核心业务图标之外，让模型能识别出新的icon呢？ 在调研当中发现，实际业务图标的设计风格和一些通用图标库接近，比如Google Fonts 、[Flaticon Design 等。

    这些图标库语义丰富，素材经过了大量使用场景的验证，是优质素材。

    由于发现实际工程内的文件名基本覆盖图标代表含义，采用了基于文件名的弱监督标注方法进行处理，从文件名中提取描述作为Florence-2 Caption。

    同样地，为了丰富训练素材，我们进行背景颜色替换，构建了light、dark两类成对变体，匹配实际业务中可能出现的UI结构变化， 比如深色/浅色模式的主题切换、iOS/Android/Web的多端视觉规范差异。 这相当于提前为模型做了“视觉分布漂移”训练，让它适应变化。

    3.3 痛点：“误报率”居高不下

    在初期训练中发现，如果只用核心图标训练，模型会倾向于“误报”，即把非图标识别成图标。 尤其是移动端业务中存在复杂图标，比如歌曲封面、专辑封面、视频封面、平台logo等。

    此类图标往往背景复杂且可能包含文字，模型会将其识别成核心图标库中的图标，从而误报率较高。 因此，训练使用负样本的目标是让模型学会拒绝：当它不确定或确实不是图标时，应该稳定地落到“非图标”表述上，避免错误成功。

    去哪里获取这样的负样本呢？CV中的负样本概念一般来自于“不是这个类别的东西”，比如“不是狗”，“不是猫”，“不是桌子”，“不是椅子”等。

    具体到Florence-2数据集的困难负样本挖掘，我们首先想到COCO数据集 。作为业界权威计算机视觉库， 里面包含大量复杂自然图像，我们从中随机挑选了部分图片，用Florence-2模型本身进行推理，输出的结果作为负样本，构建起Florence-2 Caption。

    需要注意的是，自然图像与我们当前业务UI 场景在分布上存在显著差异，因此负样本仅用于抑制“过度自信”， 而非学习具体类别边界。实际训练数据集构建中，我们设置了较低比例的负样本，其作用更接近“拒绝校准”而非分类训练。 从使用效果来看，尽管不能完全消除误报，但对于降低误报率十分有效。

    最终，完整的Florence-2数据集，包含核心图标、通用图标、负样本三类，共计约1w+张图片，对应1w+条Caption。示例如下： 

    {
    "image": "a3093s.png",
    "caption": "filled heart like icon",
    "task_type": "<CAPTION>",
    "loss_weight": 1.0
  },

    "image": "action_arrow_circle_right_dark.png",

    四、训练框架选择：为什么使用LoRA？

    有了训练数据集，下一步就是训练流程。相比其他大模型而言，Florence-2-base是个小模型，参数量为230M。但是，全参数微调在实际工程中仍然意味着 显著更高的显存占用和更复杂的训练状态管理， 尤其是在需要频繁迭代、快速回滚和多版本对比的场景下，工程成本和风险都会被放大。

    因此优先考虑参数高效微调，在参数高效微调的各类方式中，LoRA（Low-Rank Adaptation）是应用最广泛、最高效的方式。 相比之下，LoRA 将可训练参数限制在一个明确可控的增量范围内，更符合我们对低风险、可回退的质量保障要求。

    LoRA 核心原理

    LoRA 的核心思想是：不直接修改预训练权重，而是在旁边增加一个低秩分解的增量矩阵。 以下简单说明它的核心优势：

    假设有一个原始权重矩阵 W ，维度是 d×d（例如 100×100）：

    全参数微调： - 更新方式：W' = W + ΔW， 需要训练的参数：d×d = 100×100 = 10000

    LoRA 微调： - 更新方式：W' = W + B×A（W 保持冻结） - 低秩分解： - B 矩阵：d×r（例如 100×8） - A 矩阵：r×d（例如 8×100） - r 是秩（rank），通常取 4~16，远小于 d - 需要训练的参数：d×r×2 = 100×8×2 = 1600

    对比下来，假定d=100, r=8, 参数量降低倍数：10000 / 1600 ≈ 6.25 倍

    LoRA 相关知识详见： https://github.com/microsoft/LoRA

    因此，采用 LoRA 的优势可以总结为： - 小杠杆撬动大调整：把变化限制在一小块可训练增量里，基座能力不受破坏 - 回滚可控友好：LoRA适配器可以独立加载/卸载，出现问题时秒级回退

    五、训练痛点：过拟合与欠拟合中间如何平衡？

    确定了数据集、使用框架，已知的LoRA超参也不多，看起来有手有机器，启动训练后调一调参数坐等结果就行了。

    但大多数时候得到的推理效果并不理想，要么过拟合，要么欠拟合，明明损失已经降到很低了为什么实际推理起来大量对不上。 距离图标识别可交付仍有差距，为什么呢？

    早期训练时我们也遇到过这种情况，为了提升识别准确率，我们从训练参数、损失计算、训练过程等方面展开系列治理。

    5.1 分层微调：语义映射

    在常规的多模态指令微调中，针对 target_modules 的选择，大部分工作优先对语言模型 Attention 层的 qkv 参数进行训练。 这一点较好理解：有监督学习通过构建“图像 + 指令 → 目标描述”的训练样本，使语言模型在 Attention 机制中学习如何对齐视觉 token 与指令语义，从而避免在生成阶段出现随意或偏离指令的输出。

    然而，在仅微调语言模型 Attention 层的初期实验中，我们发现该策略在业务核心图标识别任务上存在明显瓶颈。具体表现为以下典型失败模式：

    失败类型 case 现象
    相似形状混淆 播放 vs 下一首 均被识别为 "media play icon"
    品牌 Logo 混淆 酷狗 LOGO vs 酷我 LOGO 均被识别为 "KuWo Music App icon"

    在该阶段，核心图标识别准确率仅约 50%–60% 区间，相似图标之间存在显著混淆，甚至视觉上差异较大的图标也会被误识别。

    进一步分析发现，预训练视觉塔（如 ViT / DaViT）主要基于自然图像数据（人物、动物、场景）进行训练，其底层与中层特征更偏向连续色调、自然纹理和复杂结构；而 UI 图标则具有高对比度、扁平化、强几何结构、符号化语义等明显不同的分布特性。

    因此，我们在保持视觉 Attention 冻结的前提下，引入了对视觉 FFN 层的微调。 在这一过程中，关键问题不在于“关注图像的哪个区域”，而在于如何将已有的自然图像特征，重新映射到高度抽象、符号化的 UI 语义空间中。

    FFN 承担了视觉表征的非线性变换与重参数化角色。通过在视觉模型全层 FFN 中引入 LoRA，可以在不破坏预训练注意力结构的前提下，实现对 UI 域特有视觉模式（如高对比度、几何结构、符号化形状）的有效适配。

    在当前阶段的配置中：

    语言模型：全量微调 Attention 与 FFN，以确保指令理解与跨模态生成能力；

    视觉模型：对所有层的 FFN（fc1 / fc2）施加 LoRA，用于完成视觉特征到语义空间的重映射。

    在这一阶段，模型主要完成从 “A circle with a triangle in the middle” → “play button icon” 的初步语义映射学习。

    为配合分层策略，不同模块采用不同学习率：

    语言模型层：lr = 2e-5

    视觉模型层：lr = 2e-6

    比例约 1:10

    控制这一学习率差异，并非假设视觉特征“无需调整”，而是控制其更新幅度，使模型优先收敛于稳定的语义对齐路径。

    5.2 Loss 权重差异化设置：明确告诉模型"哪些错误更不可接受"

    前文给出的训练集示例中，不同类型的图片对应设置了不同的 loss_weight，这里对应不同的训练策略。

    权重配置的业务逻辑

    根据图标优先级和风险等级，我们设置了三档权重：

    数据类型 loss_weight 设计出发点 风险控制
    核心图标 1.0 最重要，不能出错 误识别会导致用例失败
    通用图标 0.9 供泛化，但不能主导，影响主链路 防止海量通用数据淹没核心业务特征
    负样本 1.2 降低误报的关键 误报会导致"错误成功"，质量风险更高

    基于权重的损失计算，首先计算 token 级别的 CrossEntropy 损失，然后应用样本权重进行广播乘法，最后只对非 padding 部分计算加权平均。这种设计明确告诉优化器"哪些错误更不可接受"。

    高权重样本的错误会产生更大的梯度，驱动优化器更积极地修正这些样本的预测，从而降低误报率。

    权重调整的实战经验

    实际上，Precision（查准率）和 Recall（查全率）往往是跷跷板。 将负样本权重设为最高的 1.2，提高 Precision，极力压制误报，必然会带来漏报（降低 Recall）增加的风险。

    在自动化测试场景下，两者的代价是不对称的：

    误报（False Positive) 带来的是“阻断性”： 比如模型把一个“歌曲详情按钮”识别为“歌曲推荐按钮”。 Agent 执行点击操作后，本应拉起弹窗却跳转到了歌曲详情。 带来的后果是：测试流程跑偏，后续步骤全部失效，最终断言失败。

    漏报（False Negative) 的代价是“可处理”的： 比如模型没认出某个图标，Agent 无法执行点击。

    在实际业务使用中，OmniParser的解析逻辑会优先对图像OCR处理，在此之外的图标才会进入Florence-2图标识别，就目前使用经历来看，漏报问题基本不存在，风险可控。

    初期训练中误报较多，对于复杂图标或非图标，模型倾向于"过度自信"，通通识别成训练图标。 引入负样本权重后，误报率降至5%，对于复杂图像，模型会在输出 "Not an icon"后进一步展开图形描述，仍具备官方模型的描述能力。

    5.3 梯度处理：小数据集的精细化训练策略

    batch_size参数为物理批次大小，可以简单视为训练速度控制器，考虑到设备MPS的训练环境，我们采用了参数组合： - batch_size: 2 - 物理批次大小 - gradient_accumulation_steps: 2 - 梯度累积步数 - 有效 batch_size = 2 × 2 = 4

    Florence-2-base (230M参数) + LoRA (5.3M可训练参数) + 384×384 图像，batch_size=2 内存友好。 而小 batch 带来的梯度噪声可以避免过早收敛到局部最优。

    在训练循环中，采用： 1. 损失缩放：将每个 batch 的损失除以累积步数后再反向传播（loss = loss / accumulation_steps; loss.backward()），确保累积后的梯度幅度与直接使用大 batch 一致 2. 梯度裁剪：使用 max_norm=1.0 防止异常样本导致的梯度爆炸 3. 跳过无效批次：当某个 batch 返回空答案或无效数据时，不计入累积计数，避免污染梯度

    5.4 Warmup 策略：平滑启动避免震荡

    采用 warmup_ratio = 0.05，即前 5% 步数用于预热，配合 Cosine 衰减调度器：

    • 避免初期震荡：从 0 线性增长到目标学习率，防止大梯度破坏预训练权重
    • Cosine 衰减：后期平滑降低学习率，帮助模型收敛到更优解

    以 10873 样本为例，从训练效果来看已经足够避免初期震荡： - 总训练步数：约 40785 步（15 epochs） - Warmup 步数：约 2039 步

    5.5 早停/断点续训/Wandb：让训练过程可控可恢复

    除了核心的训练参数，实际训练流程也考虑稳定性和可恢复性。

    5.5.1 早停机制

    为了避免过拟合和资源浪费，参数patience很有必要， 一般基于经验值设定，修改为3~5轮，核心原理为在每个 epoch 结束后，计算验证集loss，如果验证loss 连续 patience 个 epoch 没有改善，自动停止训练 在我们的训练中，通常在第 10+ epoch 达到最佳验证 loss，早停机制有效避免了后续的性能退化。

    5.5.2 断点续训

    训练过程中可能因为各种原因中断（如机器重启、手动停止等）。我们实现了完整的断点续训机制：

    保存内容： 1. LoRA 适配器权重：adapter_model.bin 2. 优化器状态：optimizer_scheduler.pt 3. 训练状态：training_state.json - 当前 epoch - 最佳验证 loss - patience 计数器 - wandb run_id

    整体上，基于最佳模型目录判断是否续训练 → 加载 LoRA 适配器权重 → 恢复优化器和调度器状态、计算剩余训练轮数 → 重新初始化学习率调度 → 继续训练

    5.5.3 Wandb 指标上报：让训练过程可观测

    在微调场景中，训练的可观测性有助于辅助判断训练是否成功，最直观的就是训练/验证损失下降曲线，比如常见的过拟合判断。 我们使用Wandb 进行指标追踪，确保每次训练都有据可查。

    通过上报损失、学习率等指标， 在Wandb Dashboard 可以直观看到： - 训练/验证损失曲线是否平滑收敛 - 学习率调度是否符合预期 - 不同参数组的梯度分布是否健康

    同时，看板内记录多次训练的指标留存，也有助于事后分析解释为什么成功或失败，为参数调优提供数据支撑。

    六、把微调纳入质量体系：训练只是中间环节

    如果站在训练视角，训练完成，OminiParser逻辑改造一下，整体流程就结束了。

    但这只是一个里程碑，距离真正可交付还需要三件事：度量、灰度、回流。

    6.1 可度量：基于业务图标集对训练做Benchmark

    结合训练集样本与实际业务，我们建设了评估集：从不同主题变化（深色/浅色）、不同颜色背景覆盖核心业务图标、通用图标， 明确 In-Domain（训练集出现过的图标变体）和 Out-of-Domain（完全没见过的全新图标）图标， 相似图标对、非图标干扰。

    评估指标也不仅仅是“准确率”，包括： - 准确率：识别重点 - 误报率：对应错误成功风险。 - 稳定性：同一输入多次推理输出是否一致。

    这些指标形成“准出关卡”，每次训练产物才认为是可交付的。

    6.2 可灰度：避免大范围影响

    从服务角度，灰度能力的作用无须赘述。我们对OmniParser服务部署做了改造，新训练产物不会直接全量上线，而是：

    • 小流量验证：先在一台机器上试运行，观察误报率、识别准确率等核心指标
    • 快速回滚：LoRA 适配器的加载/卸载是秒级操作，一旦发现问题可立即切回基线版本

    这种灰度能力让我们免于担心"一次更新，全盘皆输"。

    6.3 可回流：失败样本收集形成回流

    从一个训练系统的长期演进过程来看，训练绝非定期更新训练素材，更新LoRA适配器。收集巡检中的失败用例，分析错误原因， 形成回流，才能有效解决我们实际业务中的问题。

    措施包括：

    • 把"误报的非图标"加入负样本。
    • 把"新出现的图标风格"加入通用图标。
    • 把"业务关键但经常混淆的图标对"提升为核心样本。

    七、总结：从"Demo"到"可交付"

    近两年模型的快速迭代，容易给人一种错觉：模型足够强，接入即可解决问题。

    但在真实落地中我们发现，这种Demo 幻象往往掩盖了系统层面的复杂性。

    借助 OmniParser，我们确实跨过了“看见 UI 结构”的门槛；但在随后的图标识别实践中逐渐意识到：

    模型不再是黑盒能力，而只是系统能力的一部分。

    要实现可交付的效果，必须围绕模型构建完整的系统能力，包括稳定性、可控性、可回归性与可演进性。

    这种方法并不局限于 GUI Agent，任何高频、多模态、且要求稳定语义输出的场景（如告警归因、日志治理等），本质上都面临同样的问题：

    模型并非越强越好，而是越可控，系统才越可交付。

    最终交付的并不是一个更大的模型，而是一条可以持续迭代的质量闭环。
